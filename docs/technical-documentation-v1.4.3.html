1. Project Overview

This is a Tic-Tac-Toe HTML5 game designed for publishing on CrazyGames.com, the world's leading HTML5 game platform. The game features:

  • Single Player Mode — Classic local Tic-Tac-Toe with turn indicator and win/draw detection
  • Multiplayer Mode — Real-time room-based multiplayer via WebSocket relay server
  • Wager System — Poker-style blind bid wager where both players bid secretly; lower bid wins
  • Coin Economy — Persistent coin balance (cloud-synced via CrazyGames SDK), starting with 500 coins
  • Rematch System — Rematch requests with accept/reject modal, preserving the room
  • Ad Integration — Midgame ads (interstitial) and rewarded ads for earning coins
  • Responsive Design — Dark theme, mobile-friendly, touch-optimized
What is CrazyGames? CrazyGames is a free online gaming platform that hosts HTML5 games. Game developers upload their games, and CrazyGames handles distribution, monetization (ads), and providing SDK features like user accounts, cloud saves, and multiplayer matchmaking. Developers earn revenue based on ad impressions.

4.2 Module Dependency Chain

Scripts load in this specific order (bottom of <body> in index.html):

sitelock.js → crazygames-sdk-v3.js (CDN) → crazygames.js → wager.js → game.js → multiplayer.js → ui.js → main.js
  • sitelock.js — First to load. Checks domain. If unauthorized, redirects to CrazyGames immediately. Nothing else runs.
  • crazygames-sdk-v3.js — External CDN script. May fail to load (CORS); our code handles this gracefully.
  • crazygames.js — Wrapper around SDK. Provides fallback behavior when SDK isn't available (dev mode).
  • wager.js — Coin system. Depends on CG.saveData()/CG.loadData() for persistence.
  • game.js — Pure logic. Zero dependencies on other JS files. Can run in complete isolation.
  • multiplayer.js — Networking. Depends on nothing but the browser's WebSocket/localStorage APIs.
  • ui.js — DOM manipulation. Defines display object with all DOM references at load time.
  • main.js — Last to load. Depends on ALL other modules. Initializes everything and wires event handlers.

7.3 How Win Detection Works

function checkWin(player) {
    for (const pattern of WIN_PATTERNS) {
        // pattern = [0, 1, 2] (top row), etc.
        if (pattern.every(idx => board[idx] === player)) {
            return pattern;  // Return the winning pattern (for glow effect)
        }
    }
    return null;  // No win
}

7.4 Move Validation (makeMove)

The makeMove function performs strict validation:

  1. Game must not be over
  2. Player must be the current player (prevents out-of-turn moves)
  3. Cell index must be 0-8
  4. Cell must be empty (not taken)
  5. After placing: check win → check draw → switch turn

7.5 applyRemoteMove vs makeMove

applyRemoteMove is used for network-synced moves. It skips turn validation because the server already validated whose turn it is. Just places the mark and updates state.

8.3 How Screen Transitions Work

function showScreen(name) {
    // Remove 'active' from ALL screens (hides them via CSS)
    Object.values(screens).forEach(s => s.classList.remove('active'));
    // Add 'active' to target screen (shows it via CSS)
    if (screens[name]) screens[name].classList.add('active');
    currentScreenName = name;
    // Toggle coin HUD visibility
    const showHud = ['lobby', 'game', 'wager', 'gameover'].includes(name);
    display.coinHud.classList.toggle('hidden', !showHud);
}

The CSS handles the rest: .screen { display: none; } and .screen.active { display: flex; }

8.4 How Board Rendering Works

function renderBoard(board, winLine) {
    const cells = display.board.querySelectorAll('.cell');
    const winSet = winLine ? new Set(winLine) : null;

    cells.forEach((cell, i) => {
        const mark = board[i];
        cell.textContent = mark || '';
        cell.className = 'cell';

        if (mark) {
            cell.classList.add('taken', mark.toLowerCase());
            if (winSet && winSet.has(i)) cell.classList.add('win');
            else if (winLine) cell.classList.add('loser');
        } else {
            cell.setAttribute('data-ghost', ghostPlayer);
        }
    });
}

Each cell gets CSS classes: taken (can't click), x or o (for color), win (gold glow), loser (dimmed). Ghost preview uses data-ghost attribute with CSS ::after.

9.5 Blind Bid Wager Flow (Server-Side)

The actual bid resolution happens on the server, not in wager.js. The wager module only manages the coin balance. The server handles:

  1. Both players send place_bid with their amounts
  2. Server stores both bids
  3. When both bids are in, server calculates: finalWager = min(hostBid, guestBid), pot = finalWager * 2, bonus = (hostBid === guestBid)
  4. Server sends bid_reveal to both players with all bid info
  5. Players click "Start Game" → server sends bid_start with wager/pot
  6. Wagers are deducted via Wager.deductWager()

9.6 Veto Mechanism

Either player can veto the bid result (before clicking "Start Game"). A veto starts a free game (no wager deducted). The veto is communicated via veto_bid / bid_veto messages. On rematch, the bid state is fully reset.

10. Multiplayer Module (multiplayer.js)

This is the most complex module — it handles room creation, joining, and all WebSocket/localStorage networking with a dual-mode architecture.

10.1 Dual-Mode Architecture

The module operates in two modes:

  • WebSocket mode (production): Connects to the Node.js server for real-time communication between different browsers/machines
  • Local mode (dev/fallback): Uses localStorage events for cross-tab communication on the same browser (no server needed)

Mode selection is automatic:

function connect() {
    return new Promise((resolve) => {
        // Try WebSocket first
        try { ws = new WebSocket(WS_URL); }  // ws:// + window.location.host
        catch (e) { startLocalMode(...); return; }

        // 3-second timeout — if WS doesn't connect, fall back to local
        const timeout = setTimeout(() => {
            if (!resolved) startLocalMode('timeout');
        }, WS_TIMEOUT);  // 3000ms

        ws.onopen = () => { useWS = true; done('websocket'); };
        ws.onerror = () => { startLocalMode('websocket error'); };
    });
}

10.3 Event System (Pub/Sub)

Multiplayer uses its own event system:

function on(event, callback) {
    if (!listeners[event]) listeners[event] = [];
    listeners[event].push(callback);
}

function emit(event, data) {
    if (listeners[event]) {
        listeners[event].forEach(cb => {
            try { cb(data); } catch (e) { console.error('emit error:', e); }
        });
    }
}

This allows main.js to listen for events like roomCreated, roomJoined, opponentJoined, bidReveal, etc.

10.5 WebSocket Message Handling

Incoming WebSocket messages are routed through handleWSMessage(msg) which maps server message types to local events:

function handleWSMessage(msg) {
    switch (msg.type) {
        case 'room_created':
            myRoom = msg.roomId;
            mySymbol = 'X';
            resetWager();
            emit('roomCreated', { roomId: msg.roomId, symbol: 'X' });
            break;
        case 'room_joined':
            myRoom = msg.roomId;
            mySymbol = msg.symbol;
            resetWager();
            emit('roomJoined', { roomId: msg.roomId, symbol: msg.symbol,
                                 hostName: msg.hostName, hostBalance: msg.hostBalance });
            break;
        // ... 15+ more cases ...
    }
}

10.6 Local Mode Message Handling

In local mode (no WebSocket), messages are passed via localStorage events:

function localSend(msg) {
    if (!myRoom) return;
    msg.sender = myName;
    msg._ts = Date.now();
    const key = STORAGE_PREFIX + myRoom + '_' + msg._ts;
    localStorage.setItem(key, JSON.stringify(msg));
    setTimeout(() => localStorage.removeItem(key), 5000);  // Auto-cleanup
}

function handleLocalMessage(msg) {
    // Only process messages for our room
    if (msg.roomId !== myRoom) return;
    switch (msg.type) {
        case 'join_request':    // Guest → Host
        case 'join_accept':     // Host → Guest
        case 'wager_set':       // Host → Guest
        case 'wager_confirm':   // Guest → Host
        case 'move':            // Both directions
        // ... etc
    }
}

18. Common Errors & Debugging

18.1 "Multiplayer.send is not a function"

Cause: The send function in multiplayer.js was not exported in the public API return object.

Fix: Added send to the return object: return { connect, on, off, send, ... }

Lesson: When adding new public methods to a module, always add them to the return object at the bottom of the IIFE.

18.2 "Cannot set properties of null (setting 'textContent')"

Cause: A DOM element reference stored in the display object at module load time was null. This happens when:

  • The browser caches an old version of the JS file
  • The element doesn't exist in the HTML
  • The script loads before the DOM is ready

Fix: Added null checks before accessing display.* properties. Added Cache-Control: no-store headers to prevent caching.

Lesson: Always add defensive null checks when accessing DOM elements, especially in the wager screen which has many elements.

18.3 "SyntaxError: Identifier 'username' has already been declared"

Cause: Duplicate const declarations in the same function scope (from a bad edit that duplicated variable declarations).
